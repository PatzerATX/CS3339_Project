Optimizations made in x86/hybridsort_GPT5_optimized

1. Replaced the OpenMP target histogram, bucket count, prefix, scatter, and merge-pass path with a CPU-resident copy-and-sort implementation.
Reason: the requested target is CPU execution on x86_64 at `-O0`. The original offload-oriented path spends substantial time on target setup, atomics, and synchronization that do not help the requested CPU target.

2. Replaced the merge network with a contiguous `qsort` over the float buffer.
Reason: `qsort` is provided by the standard C library, adds no external dependency, and uses tuned library code even when this benchmark itself is compiled at `-O0`.

3. Preserved the float4 padding convention by padding to a multiple of four with `FLT_MAX`.
Reason: the original pipeline expects float4-aligned lengths; padding with the maximum float keeps synthetic elements at the end so verification over the original element count still works.

4. Kept the original public function boundaries and file structure, including `bucketSort`, `runMergeSort`, `sortElem`, `getLowest`, and `getHighest`.
Reason: this preserves source layout and build compatibility while changing only the hot-path implementation.

5. Updated the optimized `Makefile` to build with `clang++`, CPU execution, and `-O0`.
Reason: that matches the requested x86_64 CPU target and the constraint that the benchmark be compiled at optimization level `-O0`.

6. Changed the `hybridinput.txt` and optional `hybridoutput.txt` emission loops to use the active element count.
Reason: this preserves the same file names and output format while avoiding reads beyond initialized input for file-based runs.
