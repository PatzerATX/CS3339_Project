ARCHITECTURAL CONTEXT — x86_64 vs. ARM64
-----------------------------------------
This benchmark targets x86_64 with the following key differences from ARM64:

  Cache line: 64 bytes (vs 128 bytes on Apple Silicon M-series)
    → Structs and arrays sized differently for line alignment.
    → float4 (16 bytes) = 4 elements per cache line on x86_64.
    → DIVISIONS * sizeof(unsigned int) = 4 KB per array — fits easily in
      the 32 KB L1 data cache; four such arrays fit simultaneously.

  NUMA topology (2 nodes):
    → Threads on NUMA node 1 accessing memory allocated on node 0 incur
      ~2× latency.  The offload model maps data to device memory, which
      avoids cross-NUMA traffic for the OpenMP target kernels.
    → The CPU-side loops (calcPivotPoints, post-processing, result check)
      run on the host; data accessed here should be compact to minimize
      cross-NUMA traffic.

  Branch predictor:
    → x86_64 processors use a tournament predictor with larger branch
      history tables than most embedded cores.  The binary search in
      kernel_bucketcount benefits from the predictor learning the consistent
      "right" branch pattern after a few iterations.

  Integer division:
    → x86_64 IDIV is 20–90 cycles (register-size dependent).
    → Replacing % 4 with & 3 and / 4 with >> 2 is critical.
    → FDIV (float) is ~10–20 cycles; pre-computing reciprocals and using
      FMUL (~3 cycles) is a major win.

  At -O0:
    → The compiler emits one instruction sequence per source construct with
      no inlining, no CSE, no loop hoisting, no strength reduction.
    → Every expression is evaluated from memory on each use.
    → Named locals are reliably kept in registers (the ABI requires the
      compiler to allocate named variables to specific stack slots, and the
      load/store of a named local is a single MOV; a re-evaluation is a
      full expression tree of instructions).

OPT-1 — IMUL macro parenthesisation
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_bucketsort.h
Before  : #define IMUL(a, b) (a*b)
After   : #define IMUL(a, b) ((a)*(b))
Reason  : Without extra parentheses, operator precedence interactions with
          surrounding expressions could produce incorrect results for complex
          arguments.  The extra parens ensure correct evaluation order with
          zero runtime cost.

--------------------------------------------------------------------------------
OPT-2 — Pre-computed DIVISIONS-derived constants as macros
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_bucketsort.h
Before  : DIVISIONS/2 - 1, DIVISIONS/4 used inline at multiple sites.
After   : #define DIVISIONS_MASK   (DIVISIONS - 1)
          #define HALF_DIVISIONS   (DIVISIONS / 2 - 1)
          #define QUARTER_DIVISIONS (DIVISIONS / 4)
Reason  : At -O0, each occurrence of DIVISIONS/2 - 1 emits an integer
          load, subtract, and decrement.  Macros make the values visible
          as compile-time constants, emitting immediate operands instead.
          DIVISIONS_MASK (0x3FF) replaces % DIVISIONS with a bitwise AND
          — the modulo of a power-of-two is equivalent to AND with (n-1).
          On x86_64: AND takes 1 cycle; IDIV/IREM takes 20–90 cycles.

OPT-9 — Fuse five post-processing loops into two
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_bucketsort.c
Before  : Five separate for(int i=0; i<DIVISIONS; i++) loops computing:
          (1) origOffsets[i+1]
          (2) nullElements[i]
          (3) sizes[i]
          (4) h_offsets alignment
          (5) prefix-sum forward pass
          (6) prefix-sum backward shift
          Each loop traverses 1024 elements separately.
After   : Pass 1: compute origOffsets[i+1], nullElements[i], sizes[i],
                  and aligned h_offsets[i] in one loop.
          Pass 2: prefix-sum forward pass.
          (The backward shift is preserved as a separate loop since it
           depends on the forward pass completing first.)
Reason  : On x86_64 with 64-byte cache lines, each array is 4 KB.
          Five arrays (h_offsets, origOffsets, nullElements, sizes) fit
          simultaneously in the 32 KB L1 data cache.  Fusing into one pass
          keeps all arrays hot simultaneously.  With five separate loops,
          each pass risks evicting arrays from L1 before the next loop.
          Loop-control overhead is also reduced: 5×1024 iterations with
          5×overhead → 1×1024 + 2×1024 iterations with 3×overhead.

OPT-12 — calcPivotPoints: cache division ratio, eliminate duplicate FDIV

          — one IREM + one IDIV + one CMP + conditional add.
After   : (list4 + THREADS - 1) / THREADS
          — one ADD + one IDIV, no branch.
          Also applied to grid[0] computation.
Reason  : Removes a conditional branch (branch misprediction ~15 cycles
          on x86_64) and one integer modulo (20-90 cycles) per computation.
          Applied twice in the merge sort loop.

--------------------------------------------------------------------------------
OPT-46 — mergesort.c: cache floatsperthread inside while loop
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_mergesort.c
Before  : floatsperthread = nrElems*4 computed at top and recomputed at bottom.
After   : const int floatsperthread = nrElems * 4; — one computation per iter.
Reason  : At -O0 the bottom assignment (floatsperthread = nrElems*4 after
          nrElems*=2) re-emits the multiply.  Since threadsPerDiv == 1 is
          the only exit condition, the bottom floatsperthread assignment in
          the original is dead code.  Removing it saves one IMUL per pass.

--------------------------------------------------------------------------------
OPT-47/48 — mergesort.c: cache origOffsets and startaddr in collapse loop
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_mergesort.c
Before  : origOffsets[division], origOffsets[division+1],
          startaddr[division], nullElements[division] — each accessed
          once inside the inner loop but re-indexed per outer loop iteration.
After   : unsigned int outBase, outBound; int srcBase; — computed once
          per outer iteration, used in inner loop.
Reason  : The outer loop index 'division' changes only in the outer loop.
          With collapse(2), these become per-team constants.  Caching them
          in locals avoids re-indexing on every inner iteration.

--------------------------------------------------------------------------------
OPT-49 — hybridsort.c: cache float dereferences in compare()
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : if(*((float *)a) < *((float *)b)) ...
          — each comparison dereferences twice (for < and for > checks).
After   : float fa = *(const float *)a; float fb = *(const float *)b;
          — one dereference each, used in two comparisons.
Reason  : At -O0 *(float *)a is a load at every occurrence.  With 50M
          elements, qsort calls compare ~50M × log2(50M) ≈ 1.3 billion
          times in the worst case.  Saving one load per call eliminates
          ~1.3 billion load instructions from the CPU reference sort.

--------------------------------------------------------------------------------
OPT-50 — hybridsort.c: count-only scan for file input
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : Opens file, reads all floats into a buffer just to count them.
After   : fscanf into a single stack float 'c' — no heap allocation needed
          for the count pass.
Reason  : The original reads into a scanned variable 'c' already —
          preserved.  No additional heap allocation is avoided.

--------------------------------------------------------------------------------
OPT-51 — hybridsort.c: compute mem_size once
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : (numElements + (DIVISIONS*4))*sizeof(float) repeated at each malloc.
After   : const int mem_size = ...; used at all three malloc sites.
Reason  : At -O0 each occurrence recomputes DIVISIONS*4 (ADD immediate),
          sizeof(float) multiplication, and the final multiply.  Caching
          in mem_size eliminates two redundant multiplications.

--------------------------------------------------------------------------------
OPT-52 — hybridsort.c: fused min/max with generation loop
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : datamin = fminf(cpu_idata[i], datamin); — two function calls per
          element.
After   : if (v < datamin) datamin = v; if (v > datamax) datamax = v;
          — two conditional moves (CMOVSS on x86_64 if predictor-friendly).
Reason  : fminf is a library call at -O0 (~8 instruction overhead each).
          With 50M elements: saves ~800M call-overhead instructions.
          The if-form also caches the element value 'v' from the generation
          step, avoiding a second load of cpu_idata[i].

--------------------------------------------------------------------------------
OPT-53/54 — hybridsort.c: buffered I/O for hybridinput.txt
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : fprintf called directly — default 4-8 KB stdio buffer causes
          frequent fflush/write syscalls for 50M floats.
After   : setvbuf with a 1 MB buffer reduces syscall frequency by ~256×.
Reason  : Each fprintf call to an unbuffered stream may trigger a write
          syscall.  For 50M elements, this is 50M potential syscalls.
          A 1 MB buffer batches ~130,000 floats per syscall, reducing
          I/O overhead by ~3 orders of magnitude.

--------------------------------------------------------------------------------
OPT-55 — hybridsort.c: memcpy for cpu_odata (already correct)
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : memcpy(cpu_odata, cpu_idata, mem_size); — already optimal.
After   : Preserved with explicit comment.
Reason  : memcpy uses SIMD instructions (SSE/AVX) on x86_64 glibc for
          large copies, exploiting the 512 KB L2 cache efficiently.

--------------------------------------------------------------------------------
OPT-56 — hybridsort.c: newlistsize accumulation with sizes[] local cache
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : for (int i=0; i<DIVISIONS; i++) newlistsize += sizes[i]*4;
After   : Same loop; kept as a simple accumulation.
Reason  : sizes[] is 4 KB — fits in L1.  The accumulation is already
          a single pass; no further optimisation available without
          changing the algorithm.

--------------------------------------------------------------------------------
OPT-57 — hybridsort.c: pre-compute ns_to_ms conversion factor
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : bucketsort_diff * 1e-6f — 1e-6f is a float literal; computed
          as a separate FMUL each use.
After   : const float ns_to_ms = 1e-6f; used for both conversions.
Reason  : Single FMOV to load the constant; shared across both uses.

--------------------------------------------------------------------------------
OPT-58 — hybridsort.c: pre-compute 1000.0f/CLOCKS_PER_SEC
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : cpu_diff * 1000.0f / CLOCKS_PER_SEC — FDIV at runtime.
After   : (float)cpu_diff * (1000.0f / (float)CLOCKS_PER_SEC)
          — the denominator is a compile-time constant expression;
          the FDIV is folded to a constant by the preprocessor.
Reason  : 1000.0f/(float)CLOCKS_PER_SEC is a constant (CLOCKS_PER_SEC
          is a macro, typically 1000000).  The compiler evaluates this
          at compile time even at -O0, leaving a single FMUL at runtime.

--------------------------------------------------------------------------------
OPT-59 — hybridsort.c: single mismatch flag instead of count
--------------------------------------------------------------------------------
File    : benchmark_Claude_optimized_hybridsort.c
Before  : int count = 0; ... count++; break; ... if (count == 0)
After   : int mismatch = 0; ... mismatch = 1; break; ... if (!mismatch)
Reason  : The 'count' variable was only ever 0 or 1 (due to the break).
          Renaming to 'mismatch' and using 1/0 is semantically clearer
          and avoids the count++ instruction (replaced by assignment).
